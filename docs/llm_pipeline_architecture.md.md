### 📄 归档文档：《折姜大学模拟器》大模型双轨制流水线与缓存解耦规划方案

#### 1. 背景与目标

当前 `app/core/llm.py` 采用单一大模型兼顾“文本生成”与“JSON结构化输出”，导致大模型的角色扮演（RP）潜力和沉浸感被 JSON 格式约束所限制。同时，大模型调用逻辑与 Redis 队列（缓存进货/消费）逻辑存在深度耦合。
**核心目标：**

1. 引入 MiniMax-M2-her 专门负责提供高情绪价值、强代入感的纯文本游戏文案。
2. 将底层生成逻辑与 Redis 缓存系统彻底解耦，为后续的 Agent 自动托管模式和高并发打好基础。

#### 2. 核心架构设计：双模型接力流水线 (Dual-Model Pipeline)

针对逻辑复杂的场景（如需要判定 `energy` 和 `sanity` 增减的随机事件），采用 **“发散（RP） -> 收敛（结构化）”** 的流水线模式：

* **Step 1：剧本创作 (MiniMax-M2-her)**
* **输入**：玩家当前状态（`PlayerStats`） + `characters.json` 中的随机人设（如“高冷导师”、“卷王室友”）。
* **输出**：无 JSON 约束的纯文本微小说/对话。


* **Step 2：逻辑提取 (当前大模型 + JSON Mode)**
* **输入**：Step 1 生成的纯文本 + 系统级 JSON Schema 提取 Prompt。
* **输出**：严格符合游戏引擎解析规范的 JSON 结构（包含发生的事件、选项以及具体的数值 `effects`）。



#### 3. 业务场景分发策略 (Routing Strategy)

为了平衡 API 成本、延迟与生成效果，按业务场景进行路由：

* **重逻辑场景（随机事件/特殊抉择）**：走完整的双模型流水线（M2-her -> 结构化模型）。
* **重文案场景（钉钉消息/CC98帖子）**：降维处理，仅调用 M2-her 生成带特定分隔符（如换行符）的纯文本，由后端代码直接用 `split()` 解析入库，无需进行二次大模型提取。

#### 4. 底层重构计划：引入内容池管理器 (Content Pool Manager)

解决 `llm.py` 既当“生产车间”又当“仓库管理员”的耦合问题：

1. **拆分职责**：新建 `core/llm_generator.py`（纯粹无状态的文本与 JSON 生成 API调用），与 `services/pool_service.py`（库存管理）。
2. **异步无阻塞进货**：`PoolService` 提供 `get_event()` 给引擎调用。当检测到 Redis `game:events_pool` 队列长度低于阈值（如 < 5）时，触发 `asyncio.create_task()` 在后台隐式调用双模型流水线进行“异步进货”，不阻塞玩家的任何交互操作。

#### 5. 实施路径规划

* **Phase 1**：剥离 `llm.py` 中的 `RedisCache` 逻辑，建立独立的 `PoolManager`。
* **Phase 2**：新增 `llm_rp.py`，使用原生 HTTP 请求对接 MiniMax，实现钉钉/CC98 的纯文案降维生成。
* **Phase 3**：串联双模型，完成复杂随机事件的流水线改造。
